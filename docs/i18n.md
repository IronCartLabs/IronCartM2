# Translations (`i18n/en_US.csv`)

Magento extracts user-facing strings from two surfaces:

1. **PHP** — every `__('<source>')` / `__("<source>", ...)` call in
   `Controller/`, `Ui/`, `Block/`, `Model/`, etc. The first arg is the
   source phrase.
2. **XML** — every element under `etc/`, `etc/adminhtml/`, and
   `view/adminhtml/` whose `translate="…"` attribute names a
   translatable surface:
   - `translate="title"` → the `title="…"` attribute literal
     (`etc/acl.xml`, `etc/adminhtml/menu.xml`).
   - `translate="label"` (and combinations like `translate="label comment"`)
     → the direct-child `<label>` element text (`etc/adminhtml/system.xml`)
     OR the element's own text when the element *is* a
     `<label translate="true">…</label>` (UI-component `*.xml`).
   - `translate="comment"` (and combinations) → the direct-child
     `<comment>` element text (`etc/adminhtml/system.xml`), including
     CDATA sections.

Every distinct source phrase the scanner discovers must have a matching
row in `i18n/en_US.csv`. en_US is the source locale, so each row is just
the phrase repeated:

```csv
"Severity","Severity"
"Run scan now","Run scan now"
```

The file is RFC-4180 with comma separators and `"`-escaping (a literal
`"` inside a field becomes `""`). It is read by Magento's built-in
`Magento\Framework\Translate` collector, which uses PHP's `fgetcsv()`
defaults — no comment-prefix support, no header row.

## Why this file exists

Adobe Marketplace's EQP scanner enforces
[`MEQP2.Translation.MissingI18n`](https://developer.adobe.com/commerce/marketplace/eqp/php-code-sniffer/):
every translatable source phrase must be backed by a CSV row, even for
en_US. A missing phrase is a **hard submission blocker**.

A v0 module without an `i18n/` directory fails the EQP scanner outright;
fixing that is the only purpose of this file at the moment. Future
translations (de_DE, fr_FR, …) would be added as sibling `i18n/<locale>.csv`
files with the same source column and a translated target column.

## Adding a new translatable phrase

When you add a `__('…')` call or a new `translate="…"` element:

1. Run `php bin/check-i18n.php` from the module root. The script will
   list every source phrase that's missing from the CSV (or warn about
   orphans whose source is no longer in the tree).
2. Add a `"<source>","<source>"` row to `i18n/en_US.csv` in
   locale-aware alphabetical order. Sorting is enforced loosely (the
   checker doesn't fail on misorder) but is a soft convention to keep
   diffs minimal.
3. Re-run the checker to confirm a clean pass:

```text
i18n/en_US.csv OK — 44 phrase(s) covered.
```

The CI workflow runs the same script on every PR (see
`.github/workflows/ci.yml`'s `i18n` job); merging is blocked until the
CSV is complete.

## Implementation notes

- `bin/check-i18n.php` uses the PHP tokenizer for PHP (so it correctly
  ignores method calls like `$obj->__construct()`) and `DOMDocument`
  for XML (so CDATA + multi-line `<comment>` blocks survive intact).
- Multi-line CDATA `<comment>` blocks are collapsed to a single
  whitespace-normalised line before being compared to the CSV. That
  matches what `bin/magento i18n:collect-phrases` does at runtime, so
  CSV rows authored manually here line up byte-for-byte with what
  Magento would emit from the same source tree.
- `bin/` scripts are excluded from the scan because they're build-time
  CLI utilities (e.g. `build-manifest.php`) — never loaded by Magento at
  runtime, so their CLI-help strings are not part of the user-facing UI
  surface MEQP enforces.
